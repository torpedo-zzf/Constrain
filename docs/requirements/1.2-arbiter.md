# 1.2 仲裁器（Arbiter）

## 核心职责
在关键节点做决策——资源分配、冲突裁决、策略执行、重试判断。所有决策可审计。

## 详细需求

| 需求项 | 规格 |
|--------|------|
| 资源锁 | Redis分布式锁，支持申请/释放/续期 |
| 冲突裁决 | FIFO、优先级队列、权重轮询三种策略 |
| 重试决策 | 指数退避，可配置最大次数、退避基数、可重试异常类型 |
| 策略配置 | YAML/JSON热加载，不硬编码 |
| 可扩展性 | 插件式策略，实现Strategy接口即可 |
| 决策审计 | 记录时间、请求方、资源、决策、原因 |

## 对外接口

class Arbiter:
    async def acquire_lock(self, resource_id: str, requester_id: str, ttl: int) -> LockResult
    async def release_lock(self, resource_id: str, requester_id: str) -> bool
    async def decide_retry(self, task: Task, error: Error) -> RetryDecision
    async def evaluate_policy(self, policy_name: str, context: dict) -> PolicyResult

## 预置策略
- ResourceLockPolicy：资源锁申请与冲突裁决
- RetryPolicy：指数退避重试
- PriorityPolicy：urgent/high/normal/low四级优先级
- RateLimitPolicy：滑动窗口限流
